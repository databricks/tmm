The Build

What you're building — zero grants on the shared data

THE RUNTIME

FastAPI + OpenAI Agents SDK, deployed as a Databricks App by Genie Code running the shipped 02_Deploy_App recipe — one plain-English prompt. (No Genie needed: Run All on that notebook does the same.)

THE TOOLS — OBO

UC functions over a shared SQL warehouse + Vector Search, all called with your forwarded token (X-Forwarded-Access-Token). Scopes: sql · vector-search.

THE MEMORY

A Lakebase resource attached at deploy: every chat turn lands in managed Postgres, written by the app's own SP into a schema it owns (memory_<your-app>) — the documented production pattern.

THE LLM

Foundation Model APIs, pay-per-token — the app's only service-principal call, and it needs no grant. Swap models with one line in app.yaml.

Grants on the shared data

On TechMart data there is no service account to over-permission, audit, or leak. If you can't see a row, neither can your agent. The only thing the app owns is its own memory schema — created and owned by its SP. The security constraint is the architecture, and it's why everyone in this room, with zero admin rights, ships their own app today.

Governance, Live

Watch the mask follow you through the app

1 · CONSENT, NOT CONFIG

The first time you open your app, Databricks shows a "Permission Requested" screen: exactly two scopes — Databricks SQL and Vector Search, the two data paths your agent uses as you. (Memory needs no consent: it runs as the app's own service principal.) You see precisely what you delegate, and nothing more is possible. Click Authorize.

2 · THE REDACTION MOMENT

Then ask it: "What's the status of order ORD-10001? Include the customer's email." The order comes back — the PII comes back ***REDACTED***. The same Unity Catalog column mask you saw in the notebook fires through your deployed agent.

3 · IT REMEMBERS

Ask a follow-up that only makes sense with memory — "and when was that order placed?" The chat header shows your session id and memory schema; every turn is being stored in Lakebase under it. You'll read those exact rows after the evals.

Nobody wrote redaction logic in the app. The mask was defined once, in the catalog — governance you didn't have to build.

The Sabotage

Three bugs are hiding in the data

Your agent's code is fine. The data lies — the most common failure mode of real enterprise agents. In Module 4, go hunting:

BUG 01 · AVAILABILITY
Discontinued, still marketed

Somewhere, marketing docs pitch a product as "available for immediate purchase" that the catalog says is Discontinued. A good agent trusts the catalog. Try asking about products you saw in Module 1.

BUG 02 · WARRANTY ★
The warranty lie

Every structured source agrees on the warranty term — the catalog column, the official policy. Only the free-text marketing copy brags about something longer. Ask your agent about a warranty and watch which source it trusts.

BUG 03 · POLICY
The too-nice policy doc

An over-permissive "Customer Satisfaction Policy (Extended)" lives in the policy table itself. No prompt fixes bad data — this one's for your data engineers.

Spoiler for Module 4: your structured data is right — your agent reads the marketing copy.

The Memory · Module 5½

Your agent kept receipts — go read them

The whole time you've been chatting, your agent has been writing every turn to Lakebase — Databricks' managed Postgres — via the Lakebase resource attached at deploy, into a schema it owns (your chat header shows it). Open Compute → Lakebase, enter the project "Agent Apps Workshop Memory", open the SQL Editor, and run — with your schema name:

SELECT session_id, message_data, created_at
FROM memory_<your_app_name>.agent_messages
ORDER BY created_at DESC LIMIT 20;

THAT'S YOUR CHAT

Find the session id from your app's header — every question you asked is there, as OLTP rows, back in double-digit milliseconds. Real Postgres, living next to your lakehouse.

TWO AUTH PATTERNS

The rows were written by your app's service principal into a schema it owns — its Postgres role set up by the deploy recipe. The data tools stay on-behalf-of-you, because governance (the PII mask) must follow the human. Each where it belongs.

Production Mode

The loop is the product

JUDGES AS REGRESSION GATES

Your eval notebook is a unit-test suite for agent behavior. Package app + eval as a Databricks Asset Bundle, re-run on every prompt, data, or model change, and fail the pipeline when warranty_accuracy drops.

TRACES IN UNITY CATALOG

Production traces land governed and queryable — lineage and audit on every agent decision, debuggable in natural language ("which traces failed the warranty judge, and why?").

EVALUATE BEFORE YOU SWAP

Changing the model is one line in app.yaml. When this lab was built, one frontier model self-corrected the planted warranty bug; another failed tool-calling outright. Same agent, wildly different behavior — which is exactly why you'll leave today owning an eval harness that settles model choices with numbers instead of vibes.
